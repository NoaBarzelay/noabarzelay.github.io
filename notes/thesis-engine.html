NOTES·Thesis Engine

Thesis Engine: five decisions and what they replaced

Owning the orchestration loop, sealing bring-your-own keys, failing loud but finishing, fixing the output schema, and gating merges on evals.

Thesis Engine (the repo is codenamed Sourcer) turns an investment thesis into a sourcing report you could bring to an investment committee. You give it a thesis, a description, and your investment criteria. It writes a falsifiable litmus test (“a company qualifies iff X”) and proposes three to five solution-oriented categories, defined by who buys and what problem is solved rather than by tech layer, and you approve or edit both before anything expensive runs. Then one research agent per category fans out in parallel over web search, an adversarial critic dedups the findings and separates real signal from narrative, a scorer ranks every surviving company against your criteria, and a writer composes a long-form report plus an executive summary and next steps. Every stage hands the next one a typed contract, and progress streams to the UI as a live agent trace.

It is a clean-room generalization of a research process I run as an investor. It contains no proprietary deal data; it works entirely from public web sources plus whatever the user provides.

The interesting part of a project like this is not the pipeline diagram, it is the decisions underneath it. These are the five that carried the most weight, each with the alternative I rejected and why.

Own the loop, rent only durability

The default for a multi-agent product right now is to adopt an agent framework and let it own control flow. I went the other way. The coordinator that runs planner, researchers, critic, scorer, and writer is plain code I wrote, and each agent is a thin tool loop over the Vercel AI SDK. The one thing I rent is durability: Vercel’s free tier caps a function at 300 seconds, a heavy research run does not fit in that window, and an in-function pipeline dies mid-run or the moment the tab closes, with nothing saved. The Workflow DevKit moves the run boundary outside the HTTP request. Starting a run returns an id, each pipeline stage is a retryable step, and the same reconnectable stream that keeps the run alive doubles as the live-trace transport, so closing the tab loses nothing.

Rejected: letting a framework own the whole loop, including the SDK’s own durable agent, which routes models through a gateway that conflicts with the bring-your-own-key design below. When the framework owns the loop, I can explain what the framework does but not what my system does. Also rejected: plain in-function streaming, which turns every heavy run into a coin flip.

Bring your own key, and treat custody as a design constraint

Visitors connect their own model access and inference bills to them, so hosting the public app costs me nothing and there is no shared key to abuse. That decision only means something if key custody survives contact with the rest of the architecture. The key travels in a request header, never the JSON body, is read into request scope, and is passed only to the provider client, never into agent context or the trace. A redaction boundary scrubs key-shaped strings before anything is logged.

The durable-workflow choice above then created a specific failure. The workflow engine persists step inputs so it can replay them, which means passing the raw key as a workflow argument would write a visitor’s API key into durable storage. The design-around: the key is sealed with AES-GCM under a server-side secret before it enters workflow state and unsealed inside each step at the moment of use, so only ciphertext is ever persisted. Rejected: passing the plaintext key through the workflow (a bounded but real leak), and a shared operator key (billing and abuse problems no rate limit fully solves).

Resilient and loud

A research run fans out several agents over the open web, and some of them will fail: a timeout, a rate limit, a model refusing to answer. The failure policy is resilient and loud. The run succeeds if at least one category succeeds, and every dead category renders in the final report as an explicit “not researched” block with the reason. In the schema this is a discriminated union (ok, partial, failed), not an exception, so a failure is data the report displays rather than a crash the user debugs.

The first live end-to-end run proved the point: three category agents timed out, each rendered as an explicit failed leg, and the run still completed with a report. I then raised the per-agent timeout. Rejected: fail-fast, where one timeout throws away every other category’s finished work, and silent degradation, where the report looks complete and quietly isn’t. The second is worse, because a sourcing report you cannot trust to be complete is worth less than one that tells you exactly where it is thin.

The model fills a structure, it does not design one

Every run produces the same report shape: the same sections, the same per-category brief, the same investment-table columns. That shape is a fixed Zod schema, and the model’s job is to fill it, never to invent it. Two reports on different theses become directly comparable and exportable, and a whole class of “the output looked different this time” failures disappears.

The schema also encodes trust rules. The overall 0 to 100 score is computed in code from typed dimension sub-scores the model emits, never emitted by the model itself, so scores are reproducible. Provenance is a typed object (url, title, access date) required on every ranked company, so “every claim has a citation” is a property a test can check, not a hope. Unknowns are null, not the string “unknown,” so downstream code can branch on real absence. Rejected: letting the writer agent free-form the report. The prose is prettier, and the result is inconsistent and impossible to evaluate.

Evals as a gate, not a promise

An agent pipeline without an evaluation harness is a demo. The harness leads with deterministic checks because they are cheap, fast, and unarguable: a structural scorer asserts that every ranked company carries a citation and dimension scores, that the total is in range, that the ranked list is actually sorted, and that the litmus test is well-formed with both pass and fail examples. Nine deterministic tests cover the score computation, the key sealing (round-trip plus rejection of tampered ciphertext), and the scorer itself, and they run in CI on every push: typecheck, then lint, then test, then build, and nothing lands red. A semantic judge, a second model family scoring report quality against a golden set of theses, is designed and deferred until CI has its own key for it.

Rejected: treating evals as a roadmap item, and starting with LLM-as-judge before the deterministic layer exists, which produces scores nobody can debug.

Where it stands

Sourcer is in active development. The pipeline runs end-to-end and has been verified live: planner, parallel researchers, critic, scorer, writer, a durable run that survives disconnects, reconnect and replay. The eval harness and CI gate are in. Still ahead: the hosted deployment, saved runs with shareable links and a public gallery, observability traces, and publishing the engine as an MCP server so any MCP client can call it. The repo is private for now; it opens together with the deployed app.
